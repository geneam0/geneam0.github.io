Minesweeper Game

  • Gene Lam
  • Thursday, March 26, 2020 7:26pm

Minesweeper is a single-player puzzle video game. The objective of the game is to clear a rectangular board containing hidden "mines" or bombs without detonating any of them, with help from clues about the number of neighboring mines in each field.

I wrote this in my AP Computer Science class in my Junior year of high school. Looking back, this code is very inefficient and redundant. Nevertheless, I learned a lot from this project and enjoyed coding it. This game is played in the terminal.

The program first generates the mine field. It's easy to program in where the mines go or the rate in which they appear (20% in my code). I had a difficult time generating the numbers around the mine. Back then, I had just started learning recursive functions and had no idea what dynamic programming was so I brute-forced my way. For each box in the 2D array, the program counted all the mines surrounding it if it was a corner box, middle box, or side box.

The next step for the program was handling user input, which was easy as well. My next rip-my-hair-out moment was if the player hit a zero box. If the player hits a number, that number appears. If they hit a mine, game over. But if they were to hit a 0, the zeroes trigger the surrounding numbers to appear. This was where my recursive training had been tested. But according to my code, it seems like I brute-forced my way around that too, using a bunch of if-statements and calling the same function six times for each if-statement. *sigh* At least it works.

I remember feeling overjoyed when I finished this project. I had so many rip-my-hair-out moments from trying to figure out efficient algorithms. And when I couldn't do that, all my rip-my-hair-out moments came from writing all the if-statements. Sorry Ms. Qiu. I know you're disappointed by my 426 lines, and I hope you know that I'm cringing on my seat right now with you.
